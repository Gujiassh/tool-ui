Tool UI actions should answer one question clearly: **what can the user do next?**

Start with this model:

1. **Display surfaces** render tool output.
2. **Follow-up actions** are lightweight next steps (open, copy, retry).
3. **Decision actions** are consequential and should produce a durable result.

## Action Decision Matrix

| User interaction | Durable outcome required? | Surface | Call `addResult(...)`? |
| --- | --- | --- | --- |
| Open link, copy value, retry fetch | No | Follow-up actions | No |
| Approve/reject, publish/cancel, confirm order | Yes | Decision actions | Yes |
| View data only | N/A | Display surface only | No |

## Current API Mapping

The model above is conceptual. In the current Tool UI component APIs, most surfaces still transport actions through:

- `responseActions`
- `onResponseAction`
- `onBeforeResponseAction`

Use those props according to intent:

- **Follow-up action:** perform local/runtime side effects only.
- **Decision action:** call `addResult(...)` and re-render the receipt state (`choice`, `decision`, or equivalent).

Some decision-first components already expose dedicated callbacks (`onConfirm`, `onCancel`) instead of generic action callbacks.

## Minimal Implementation

### 1) Follow-up actions (no durable state)

```tsx
import { type Toolkit } from "@assistant-ui/react";
import { LinkPreview } from "@/components/tool-ui/link-preview";
import { createResultToolRenderer } from "@/components/tool-ui/shared";
import { safeParseSerializableLinkPreview } from "@/components/tool-ui/link-preview/schema";

const toolkit: Toolkit = {
  previewLink: {
    type: "backend",
    render: createResultToolRenderer({
      safeParse: safeParseSerializableLinkPreview,
      render: (data) => (
        <LinkPreview
          {...data}
          responseActions={[
            { id: "open", label: "Open" },
            { id: "copy", label: "Copy URL" },
          ]}
          onResponseAction={(actionId) => {
            if (actionId === "open") window.open(data.href, "_blank");
            if (actionId === "copy") navigator.clipboard.writeText(data.href);
          }}
        />
      ),
    }),
  },
};
```

### 2) Decision actions (durable result + receipt)

```tsx
import { type Toolkit } from "@assistant-ui/react";
import { OptionList } from "@/components/tool-ui/option-list";
import {
  SerializableOptionListSchema,
  safeParseSerializableOptionList,
} from "@/components/tool-ui/option-list/schema";
import { createArgsToolRenderer } from "@/components/tool-ui/shared";

const toolkit: Toolkit = {
  choosePlan: {
    description: "User chooses a subscription plan",
    parameters: SerializableOptionListSchema,
    render: createArgsToolRenderer({
      safeParse: safeParseSerializableOptionList,
      idPrefix: "plan-choice",
      render: (args, { result, addResult }) =>
        result ? (
          <OptionList {...args} value={undefined} choice={result} />
        ) : (
          <OptionList
            {...args}
            value={undefined}
            onConfirm={(selection) => addResult?.(selection)}
          />
        ),
    }),
  },
};
```

For receipt writing guidance and component-level receipt states, see [Receipts](/docs/receipts).

## Optional Decision Envelope (App-Level Convention)

When you want standardized analytics/auditing across decision surfaces, wrap decision outcomes in an envelope before calling `addResult(...)`.

```ts
type DecisionResult<TPayload extends Record<string, unknown> = {}> = {
  kind: "decision";
  version: 1;
  decisionId: string;
  actionId: string;
  actionLabel: string;
  at: string;
  payload?: TPayload;
};

function createDecisionResult<TPayload extends Record<string, unknown> = {}>(
  input: Omit<DecisionResult<TPayload>, "kind" | "version" | "at">,
): DecisionResult<TPayload> {
  return {
    kind: "decision",
    version: 1,
    at: new Date().toISOString(),
    ...input,
  };
}
```

## Shared Action Contract

```ts
type Action = {
  id: string;
  label: string;
  sentence?: string;
  confirmLabel?: string;
  variant?: "default" | "destructive" | "secondary" | "ghost" | "outline";
  disabled?: boolean;
  loading?: boolean;
  shortcut?: string;
  icon?: ReactNode; // runtime only
};

type ActionsProp =
  | Action[]
  | {
      items: Action[];
      align?: "left" | "center" | "right";
      confirmTimeout?: number;
    };
```

## Runtime Behavior (Current Implementation)

Action execution is normalized by shared `ActionButtons` logic:

- `confirmLabel` enables two-step confirmation.
- Confirmation expires after `confirmTimeout` (default `3000ms`).
- Only one action can execute at a time (execution lock).
- `Escape` clears a pending confirmation state.
- If `variant` is omitted, common negative action IDs (`cancel`, `reject`, `dismiss`, etc.) infer `ghost`; others render as `default`.

## Practical Guidance

- Keep actions tight: usually 2 to 3 buttons.
- Put consequential choices in decision surfaces and always produce a receipt.
- Keep follow-up actions reversible and local.
- Use `sentence` on actions when you want deterministic assistant narration.
- For durable outcomes, align copy and behavior with [Receipts](/docs/receipts).
